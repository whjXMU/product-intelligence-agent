# Agent 使用方式设计

## 背景

当前竞品分析工作台更接近“表单提交任务”的 Demo 形态：用户需要先按固定字段填写任务，再由 Agent 执行分析。

这种方式适合标准化任务，但不适合真实产品工作中的需求澄清、逐步讨论、局部修改、历史复用等场景。后续工作台应从“任务创建器”升级为“项目内的持续协作空间”。

## 核心判断

Agent 的交互入口不应只有固定表单。

更合理的主线是：

```text
项目空间
  -> 对话澄清
  -> Brief 确认
  -> Agent 执行
  -> 文档生成
  -> 模块讨论
  -> 用户确认
  -> 知识入档
  -> 历史复用
```

其中有两个关键确认点：

```text
需求确认：确认 Agent 是否理解对了
结果确认：确认报告和结论是否可以入档复用
```

## 使用方式一：从 0 创建完整分析或需求文档

用户从一个模糊想法开始，与 Agent 逐步讨论，最终形成完整分析或需求文档。

典型流程：

```text
用户描述模糊想法
Agent 追问澄清
生成结构化 Brief
用户确认或编辑 Brief
Agent 执行分析
生成完整需求 / 分析文档
用户审阅、修改、确认
归档为项目资产
```

适用场景：

```text
分析某个竞品
为新功能做需求拆解
判断某个产品机会
从业务想法生成 PRD 初稿
整理竞品差异和行动建议
```

建议产物：

```text
需求背景
目标用户
问题定义
竞品分析
功能范围
关键流程
验收标准
风险与待确认项
后续行动建议
```

## 使用方式二：基于已有文档的单模块深入讨论

完整文档生成后，用户通常不会一次性确认全部内容，而是围绕其中某个模块继续打磨。

典型场景：

```text
继续完善“用户画像”
重新讨论“竞品差异”
展开“核心流程”
补充“验收标准”
把“机会点”转成具体需求
把“风险点”转成方案约束
```

建议交互：

```text
用户点击文档中的某个模块
系统打开针对该模块的对话
Agent 带着当前模块、全文上下文、历史判断继续讨论
生成该模块的新版本
用户确认
替换或追加到文档
保存版本记录
```

关键原则：

```text
局部讨论不应直接污染整篇文档
模块修改应有确认动作
模块应支持版本记录
模块状态应独立管理
```

模块状态建议：

```text
draft
reviewing
confirmed
needs_update
deprecated
```

## 使用方式三：进入历史项目继续协作

历史项目不应只是报告列表，而应是一个带项目记忆的协作空间。

用户可以基于已有分析继续发起新讨论：

```text
基于上次 OpenAI vs DeepSeek 的分析，继续分析 pricing 页面
把之前的机会点 3 展开成 PRD
现在竞品改版了，帮我更新差异分析
基于历史结论，重新评估我们的首页信息架构
```

项目需要沉淀可持续上下文：

```text
项目背景
历史任务
已确认结论
已归档报告
关键 insight
未解决问题
当前需求版本
```

进入历史项目时，用户不应面对空白输入框，而应看到项目当前状态、历史结论和可继续操作的入口。

## 使用方式四：带材料进入的分析

真实工作中，用户经常已经有一些零散材料，而不是从完全空白开始。

可能材料包括：

```text
已有 PRD
竞品截图
会议纪要
用户反馈
网页链接
调研表
埋点数据
```

建议流程：

```text
上传或粘贴材料
Agent 总结材料
识别已有信息和缺口
生成结构化 Brief
用户确认
进入分析或文档生成
```

这个模式的重点是让 Agent 先整理输入，再进入执行，而不是要求用户把零散材料手动整理成表单字段。

## 使用方式五：基于模板或方法论快速启动

模板仍然有价值，但应作为快捷方式，而不是唯一入口。

可提供的模板：

```text
竞品首页分析
功能需求拆解
用户旅程分析
PRD 生成
信息架构评审
功能对比矩阵
版本规划建议
```

建议流程：

```text
选择模板
Agent 询问必要变量
生成 Brief
用户确认
执行分析
```

模板的定位是加速器，不是输入限制。

## 对产品结构的要求

为了支持以上使用方式，系统不应只围绕 Task 建模，还需要支持项目、对话、文档、模块、结论和知识沉淀。

建议核心对象：

```text
Project
Conversation
Brief
Task
Document
DocumentSection
Insight
Source
Decision
Version
```

其中 `Decision` 很重要，因为后续可复用的知识不只是 Agent 生成过的内容，而是用户确认过的判断。

示例：

```text
Decision {
  title: "首页优先突出企业级安全能力"
  source: "OpenAI vs DeepSeek 首页分析任务"
  confirmedByUser: true
  relatedSections: ["定位分析", "首页信息架构建议"]
}
```

后续新项目引用历史上下文时，应优先引用已确认的 `Decision` 和 `Insight`，避免把未经确认的 Agent 初稿当成事实。

## 建议的信息架构方向

工作台可以采用三栏结构：

```text
左侧：项目 / 任务 / 历史文档导航
中间：当前项目工作区，包括对话、Brief、执行过程、文档
右侧：上下文、资料、Trace、调试信息、产物引用
```

右侧信息应分层：

```text
普通用户：项目上下文、引用资料、关键结论、阶段产物
高级用户：执行步骤、工具调用摘要
开发者：requestId、trace、错误堆栈、raw payload
```

## 结论

Agent 的产品价值不应停留在“帮我生成一次报告”，而应升级为“帮我管理一个产品分析项目的思考过程和结论资产”。

因此后续交互重构应优先支持：

```text
自由对话输入
结构化 Brief 确认
报告与模块级审阅
历史项目继续讨论
用户确认后的知识入档
可复用的项目上下文
```
