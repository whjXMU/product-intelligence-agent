# Agent 结果形态设计

## 背景

Agent 工作台的结果不应只有一份文字报告。不同用户场景需要不同形态的产物：有的用于快速判断，有的用于团队同步，有的用于进入需求设计，有的用于长期知识复用。

结果形态应分成两层：

```text
面向理解的分析结果
面向行动的产品产物
```

## 结果形态一：比较评分和可视化

适用于竞品分析、方案对比、产品体验评估等场景。

建议流程：

```text
确认比较对象
确认比较维度
确认评分规则
Agent 执行分析
生成评分表 / 图表 / 对比矩阵
用户确认或调整评分
```

常见比较维度：

```text
信息架构
核心卖点表达
转化路径
功能完整度
用户体验
价格透明度
技术可信度
企业服务能力
```

输出形式：

```text
对比评分表
雷达图
条形图
维度优劣势矩阵
关键差异摘要
评分依据说明
```

关键原则：

```text
评分必须带依据
维度应允许用户确认和编辑
分数应允许用户调整
图表应能回溯到原始证据和文字解释
```

## 结果形态二：文字分析报告

适用于归档、团队同步、复盘和决策沟通。

报告内容可以包括：

```text
执行摘要
分析背景
竞品概览
维度对比
关键发现
机会点
风险点
建议动作
引用来源
待确认问题
```

报告应支持不同粒度：

```text
完整报告：适合归档和团队阅读
摘要报告：适合快速同步和决策
模块报告：适合围绕单个问题继续讨论
```

报告生成后不应直接归档，应进入审阅状态：

```text
draft_report
reviewing
approved
archived
```

用户应能修改标题、摘要、段落、结论、引用和报告结构。

## 结果形态三：需求建议和初稿

竞品分析后的需求建议应作为“候选需求”，而不是直接生成正式需求文档。

推荐流程：

```text
分析结果
  -> 需求机会点
  -> 需求建议列表
  -> 用户选择某些建议
  -> 进入需求确认流程
  -> 生成完善需求文档
```

候选需求字段建议：

```text
需求名称
来源洞察
目标用户
解决的问题
建议方案
预期价值
实现复杂度
优先级
风险
是否进入详细需求设计
```

用户确认选中某个候选需求后，再进入详细需求设计：

```text
需求澄清
边界确认
用户流程
交互细节
验收标准
版本范围
最终 PRD
```

关键原则：

```text
需求建议是精简候选项
正式需求文档必须经过用户确认
分析里的想法不能自动等同于已确定需求
```

## 结果形态四：关键 Insight 和决策点

知识库最需要沉淀的不是整篇报告，而是被用户确认过的关键判断。

可沉淀内容：

```text
已确认的判断
关键发现
用户认可的机会点
竞品优势和弱点
后续项目可复用的结论
```

示例：

```text
DeepSeek 首页更强调低门槛和开发者友好。
OpenAI 首页更强调平台能力和企业可信度。
如果产品面向 B 端，首页应优先强化安全、稳定性和集成能力。
```

每条 Insight 建议包含：

```text
来源任务
来源报告
证据
用户是否确认
适用范围
可信度
标签
更新时间
```

后续新项目引用历史上下文时，应优先引用已确认 Insight，避免把未经审阅的 Agent 初稿当作事实。

## 结果形态五：行动计划和 Roadmap 建议

分析完成后，用户通常还需要知道下一步做什么。

行动计划可以包括：

```text
短期优化建议
中期产品机会
长期战略方向
优先级排序
依赖条件
验证方式
风险提醒
```

这类结果介于分析报告和正式需求文档之间，适合产品负责人进行决策和排期前讨论。

## 结果形态六：可复用模板和分析框架

当一次分析完成后，系统可以把用户确认过的维度、评分规则和报告结构沉淀为可复用模板。

示例：

```text
竞品首页分析模板
AI 产品定价页分析模板
B 端 SaaS onboarding 分析模板
功能对比评分模板
```

模板可以复用：

```text
分析维度
评分规则
报告结构
必要输入字段
推荐输出形式
```

模板定位是加速器，不应成为用户输入的限制。

## 结果体系总结

建议最终结果体系分为两类。

分析型结果：

```text
比较评分 / 可视化
文字分析报告
关键 Insight
```

行动型结果：

```text
需求建议 / 候选需求
行动计划 / Roadmap 建议
可复用模板 / 分析框架
```

## 关键设计原则

```text
中间环节确认比较维度，最终结果再按维度评分。
分数、图表和表格必须能解释依据。
报告生成后需要用户审阅和编辑。
需求建议只能是候选项，不能自动成为正式需求。
用户确认后的 Insight 才适合进入知识库。
一次分析的结构和方法可以沉淀为后续模板。
```
