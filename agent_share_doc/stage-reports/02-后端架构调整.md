# 阶段 02 后端架构调整方案

## 文档性质

本文档是阶段 02 后端架构整理的确认方案。

此前后端已经完成 `analysis_tasks` 业务骨架，接口和数据库链路可用。但当前实现仍偏“快速跑通链路”，在正式接入 Agent Workflow 前，需要先补齐 Nest 后端的基础架构边界，避免后续把校验、执行、trace、错误处理、返回格式、配置读取都堆进单个 service。

本方案目标不是重写后端，而是在不改变阶段 02 核心业务行为的前提下，建立后续可持续扩展的最小后端架构。

## 总体结论

当前 `apps/api` 的模块方向是正确的：

- 已按 Nest feature module 拆出 `analysis-tasks`；
- 已有 TypeORM entity、service、controller；
- 已通过 shared DTO 建立前后端契约；
- 没有把 `packages/agent-mvp` 实验代码并入正式后端。

但继续推进前，需要补齐以下架构能力：

- 配置集中化；
- 数据库运行配置和 migration CLI 配置统一；
- 运行时请求校验，优先使用 `zod`；
- 统一错误响应；
- 统一成功响应的规则设计；
- request id、日志等基础 middleware；
- mapper、runner、validator 从业务 service 拆出；
- e2e 测试从脚手架状态升级为真实 API 验证。

## 改造原则

1. **保持外部接口业务语义不变**

   本阶段仍然保留：

   ```text
   POST /analysis-tasks
   GET /analysis-tasks
   GET /analysis-tasks/:id
   POST /analysis-tasks/:id/run-mock
   ```

2. **先建立边界，再扩大能力**

   不在阶段 02 接真实 LLM、不接队列、不接正式 Agent Workflow。

3. **公共基建只做最小闭环**

   可以先有统一错误、request id、zod 校验和响应类型定义；成功响应是否全局包裹，需要等前端一起切换。

4. **shared 负责契约，api 负责执行**

   `packages/shared` 可以承载 DTO、response 类型、zod schema；`apps/api` 承载 Nest pipe/filter/interceptor/middleware 和业务实现。

5. **不要引入无必要依赖**

   项目已确认 Agent 部分会使用 `zod` 做运行时校验，本方案也采用 `zod`。若实际 package manifest 缺少声明，应补 manifest，但不要为了架构整理引入 `class-validator` / `class-transformer` 等另一套校验体系。

## 公共基建方案

### 1. 配置集中化

当前问题：

- `main.ts` 直接读取 `WEB_ORIGIN`、`API_PORT`；
- `app.module.ts` 直接读取数据库环境变量；
- `data-source.ts` 再维护一份数据库配置；
- runtime TypeORM 和 migration CLI 配置存在漂移风险。

确认方案：

新增：

```text
apps/api/src/config/app.config.ts
apps/api/src/config/database.config.ts
apps/api/src/database/typeorm-options.ts
```

职责：

- `app.config.ts` 负责读取 `API_PORT`、`WEB_ORIGIN`；
- `database.config.ts` 负责读取数据库连接环境变量；
- `typeorm-options.ts` 负责从同一份数据库配置生成 Nest runtime options 和 TypeORM CLI options；
- `app.module.ts` 和 `data-source.ts` 不再各自拼配置。

阶段 02 暂不强制引入 `@nestjs/config`。普通 TS 函数即可，后续环境变量变复杂时再升级。

### 2. zod 运行时校验

当前问题：

- `CreateAnalysisTaskRequest` 是 TypeScript interface，运行时不存在；
- controller 直接接收 body；
- service 内手写校验，职责偏重；
- 后续 Agent 输入、result、trace 都需要运行时 schema。

确认方案：

采用 `zod` 作为统一运行时校验方案。

建议在 shared 增加 schema：

```text
packages/shared/src/analysis-task.ts
```

新增：

```ts
createAnalysisTaskRequestSchema
analysisTaskInputSchema
```

并从 schema 推导类型：

```ts
export type CreateAnalysisTaskRequest = z.infer<
  typeof createAnalysisTaskRequestSchema
>;
```

API 侧新增：

```text
apps/api/src/common/pipes/zod-validation.pipe.ts
```

controller 使用方式：

```ts
@Post()
async create(
  @Body(new ZodValidationPipe(createAnalysisTaskRequestSchema))
  request: CreateAnalysisTaskRequest,
) {}
```

注意：

- 不引入 `class-validator`；
- 不把 zod schema 写在 `apps/api` 私有目录里，避免前后端契约分裂；
- shared package 应在自身 `package.json` 声明 `zod` 依赖，保证 workspace 包依赖关系清晰。

### 3. 统一错误响应

当前问题：

- Nest 默认错误格式和未来业务错误格式不统一；
- `BadRequestException`、`NotFoundException`、数据库错误、zod 错误的响应形态会不同；
- 前端难以稳定处理错误。

确认方案：

新增全局异常过滤器：

```text
apps/api/src/common/filters/http-exception.filter.ts
```

统一错误响应格式：

```ts
export interface ApiErrorResponse {
  success: false;
  error: {
    code: string;
    message: string;
    details?: unknown;
  };
  meta: {
    requestId: string;
    timestamp: string;
    path: string;
  };
}
```

错误 code 建议先定义：

```text
VALIDATION_ERROR
NOT_FOUND
DATABASE_ERROR
INTERNAL_ERROR
```

业务异常可逐步引入：

```text
ANALYSIS_TASK_NOT_FOUND
ANALYSIS_TASK_INVALID_STATUS
```

注册位置：

```text
main.ts
```

使用：

```ts
app.useGlobalFilters(new HttpExceptionFilter());
```

阶段 02 可以先统一错误响应，不强制统一成功响应，以降低对前端的破坏。

### 4. 统一成功响应

当前 API 返回的是裸 DTO：

```json
{
  "id": "...",
  "title": "...",
  "status": "created"
}
```

长期更推荐统一为：

```ts
export interface ApiSuccessResponse<T> {
  success: true;
  data: T;
  meta: {
    requestId: string;
    timestamp: string;
  };
}
```

但这个改动会影响所有前端调用。

确认方案：

- 阶段 02 架构整理时，可以先新增 `ApiSuccessResponse<T>`、`ApiErrorResponse`、`ApiResponse<T>` 类型和 `response` 工具；
- 可以实现 `ResponseInterceptor`，但不建议立即全局启用；
- 等前端任务工作台开始对接时，由主窗口确认是否一起切换为统一包裹格式；
- 如果前端已经按裸 DTO 开发，则不要在同阶段突然全局包裹成功响应。

推荐过渡策略：

```text
当前阶段：
  成功响应继续返回裸 DTO
  错误响应先统一
  shared 中预留 ApiResponse 类型

下一阶段：
  前后端一起切换 success/data/meta 格式
```

### 5. Request ID middleware

当前问题：

- 请求没有 request id；
- 出错时无法把前端报错、后端日志和 trace 串起来；
- 未来 Agent workflow trace 也需要可关联的 request/task 执行上下文。

确认方案：

新增：

```text
apps/api/src/common/middleware/request-id.middleware.ts
```

行为：

- 如果请求头已有 `x-request-id`，沿用；
- 如果没有，生成 UUID；
- 写入 response header：`x-request-id`；
- 挂到 request 对象上，供 exception filter / logger 使用。

阶段 02 可以用 Node `crypto.randomUUID()`，不新增依赖。

### 6. 日志与观测

阶段 02 不引入完整 logger 框架。

确认方案：

- 先使用 Nest 默认 logger；
- exception filter 输出必要错误日志；
- request id 进入错误响应；
- 未来接 Agent Workflow 后，再设计结构化日志、trace id、usage、cost、step trace。

不要现在引入复杂 observability SDK。

## analysis-tasks 模块调整方案

### 1. 目录整理

从：

```text
analysis-tasks/
  analysis-task.entity.ts
  analysis-tasks.controller.ts
  analysis-tasks.service.ts
  analysis-tasks.module.ts
```

调整为：

```text
analysis-tasks/
  analysis-tasks.module.ts
  controllers/
    analysis-tasks.controller.ts
  entities/
    analysis-task.entity.ts
  services/
    analysis-tasks.service.ts
    analysis-task-mock-runner.service.ts
  mappers/
    analysis-task.mapper.ts
```

目的：

- HTTP 边界、持久化模型、业务编排、执行器、DTO 映射分开；
- 后续加入正式 `AnalysisTaskWorkflowRunnerService` 时，不挤进单个 service 文件。

### 2. Service 瘦身

`AnalysisTasksService` 只保留应用用例编排：

```text
create
findAll
findOne
runMock
```

职责：

- 调 repository；
- 协调状态流转；
- 调 runner；
- 调 mapper；
- 抛业务异常。

不再负责：

- 生成 mock report 细节；
- 手写请求 body 校验；
- DTO 字段映射细节。

### 3. Mock runner 独立

新增：

```text
analysis-task-mock-runner.service.ts
```

职责：

```ts
run(task: AnalysisTaskEntity): {
  result: AnalysisTaskMockResult;
  trace: AnalysisTaskTrace;
}
```

注意：

- runner 不处理 HTTP；
- runner 初期不直接写数据库；
- service 负责状态持久化；
- 后续正式 Agent Workflow runner 可以沿用同样接口。

### 4. Mapper 独立

新增：

```text
analysis-task.mapper.ts
```

职责：

```ts
toAnalysisTaskDto(entity): AnalysisTaskDto
toAnalysisTaskListItemDto(entity): AnalysisTaskListItemDto
```

目的：

- 避免 service 充满字段搬运；
- 未来 result / trace schema 变化时集中调整；
- 便于单独测试。

### 5. 状态流转与失败处理

当前 `runMock` 成功路径可用，但失败时可能停在 `running`。

确认方案：

改造为：

```text
find task
mark running
try:
  runner.run(task)
  mark completed with result/trace
catch:
  mark failed with errorMessage/trace
  throw normalized exception
```

阶段 02 暂不引入事务和队列。

后续正式 workflow 再考虑：

- 防重复执行；
- 乐观锁；
- 任务取消；
- 队列；
- 长任务心跳；
- workflow step trace。

### 6. 参数校验

controller 的 `:id` 参数应加 UUID 校验：

```ts
@Param('id', new ParseUUIDPipe()) id: string
```

非法 id 应返回稳定的 validation error，而不是下沉到数据库层。

## API 响应格式决策

### 当前阶段的确认决策

为了不打断阶段 02 前端对接，当前阶段建议：

- 成功响应暂时保持裸 DTO；
- 错误响应统一；
- shared 中新增统一响应类型；
- response 工具和 interceptor 可以先实现但不全局启用。

### 未来推荐格式

成功：

```json
{
  "success": true,
  "data": {},
  "meta": {
    "requestId": "req_xxx",
    "timestamp": "2026-06-22T00:00:00.000Z"
  }
}
```

失败：

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid request body",
    "details": []
  },
  "meta": {
    "requestId": "req_xxx",
    "timestamp": "2026-06-22T00:00:00.000Z",
    "path": "/analysis-tasks"
  }
}
```

### 为什么不现在强制成功响应包裹

因为这会改变所有现有接口返回结构：

```text
AnalysisTaskDto
```

变成：

```text
ApiSuccessResponse<AnalysisTaskDto>
```

这需要前端同步改调用层。阶段 02 后端架构调整可以先把工具和类型准备好，等前端子窗口开始对接时统一决定。

完成本次调整后，前端任务工作台可以更稳定地接入后端。

- `AnalysisTaskMockRunnerService` 替换或并行为 `AnalysisTaskWorkflowRunnerService`；
